---

The candidate picker shell is a self-contained HTML file at `assets/candidate_picker_shell/index.html` that handles image selection when you choose the multi-candidate generation path in Stage 4. Instead of reviewing raw image files one by one, you receive a single organized interface that groups all candidates by slide page, lets you compare them, and provides a single button to copy your selections back into the chat as structured codes. This shell is only used when you explicitly opt for multiple candidates per slide — the single-candidate path skips it entirely.

## When it is used

The candidate picker shell appears at one specific point in Stage 4: after the skill generates multiple candidate images for each slide and before it assembles the final review HTML. It is not part of the preview or review stages, and it is not used on the single-candidate path.

<Tabs>
  <Tab title="Multi-candidate path">
    When you choose to generate multiple candidates per slide, the skill generates the requested number of candidates for each page using the same approved page prompt for that slide. All candidates are assembled into the picker shell, the shell is opened, and the workflow pauses until you return your selection codes.

    The skill does not proceed to the review HTML until you have pasted your selection codes into the chat.
  </Tab>
  <Tab title="Single-candidate path">
    When you choose one final image per slide, the candidate picker stage is skipped entirely. The skill moves directly from generation into the review HTML stage without opening the picker shell at all.
  </Tab>
</Tabs>

## How the selection works

<Steps>
  <Step title="Candidates assembled into shell">
    After all candidates are generated, the skill inserts them into `assets/candidate_picker_shell/index.html`. Candidates are organized by slide page so you can compare options for each page in sequence. The companion script `build_candidate_picker_html.py` assists with this population step.
  </Step>
  <Step title="Shell opened">
    The skill immediately attempts to open the local candidate picker HTML. If automatic opening fails, it explains the blocker and provides the file path as a fallback.
  </Step>
  <Step title="You review candidates per page">
    Inside the shell, you inspect the candidate images for each slide. The interface is designed to keep candidate comparison primary — images are not buried under control panels.
  </Step>
  <Step title="You click 复制全部编号">
    When your selections are ready, click the **复制全部编号** button in the shell. This copies all selection codes to your clipboard as structured output that identifies which candidate was chosen for each slide page.
  </Step>
  <Step title="You paste codes into chat">
    Return to the chat input, paste the copied codes, and send them. The skill maps each code to the corresponding selected image.
  </Step>
  <Step title="Skill maps codes to images">
    Once the codes are received, the skill resolves each selection to its chosen page image and assembles that set as the input for the final review stage.
  </Step>
</Steps>

## What selection codes are

Selection codes are structured identifiers that tell the skill which candidate image was chosen for each slide page. They are generated by the shell's **复制全部编号** button and are not something you write manually. When you paste them into the chat, the skill reads the codes and maps each one to the correct page image before proceeding.

<Tip>
  **Interaction flow summary:** generate candidates → skill opens picker shell → you compare candidates per page → click **复制全部编号** → return to chat input → paste codes → send → skill maps codes to selected images → review stage begins.

  Do not close the picker shell or skip the paste step. The skill waits for your codes before assembling the review HTML.
</Tip>

## Critical sequencing rule

If you chose the multi-candidate path, the skill does not jump from generation to the review HTML before you have returned your candidate codes. The picker shell interaction is a required checkpoint. Sending the codes is the signal that triggers the next stage.
